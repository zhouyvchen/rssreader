Day1 - Android Java RSS 阅读器学习复盘

主题：简洁 RSS 阅读器 Demo，从功能实现到问题排查。

Java Room RecyclerView XmlPullParser HttpURLConnection WebView
今日核心收获： 你开始从“代码怎么写”转向“为什么这样写、数据怎么流、什么时候刷新 UI”。 这是学习 Android 基础开发很重要的一步。

一、涉及到的知识点

1. RSS / Atom / Feed 概念

Feed 不是文章本身，而是“订阅源”。一个订阅源下面可以有多篇文章。

Feed 订阅源
  ├── Article 文章 1
  ├── Article 文章 2
  └── Article 文章 3

项目中的对应关系：

  • FeedEntity：保存订阅源信息，对应 feeds 表。
  • ArticleEntity：保存文章信息，对应 articles 表。
  • ArticleEntity.feedId：用于关联某篇文章属于哪个订阅源。

2. XML 解析流程

RSS 和 Atom 都是 XML 格式。项目中使用 XmlPullParser 解析。

  • getEventType()：读取当前解析器停留在哪个 XML 事件上。
  • next()：移动到下一个 XML 事件，并返回新的事件类型。
START_DOCUMENT
START_TAG rss
START_TAG channel
START_TAG title
TEXT Demo
END_TAG title
END_TAG channel
END_TAG rss
END_DOCUMENT

你还理解了为什么点进 getEventType() 看不到实现：XmlPullParser 是接口，具体实现由系统内部创建。

3. 网络读取

项目使用 HttpURLConnection 请求订阅源 XML。

BufferedReader.readLine() 会读取一行文本，但不会保留行尾换行符。因此代码里手动追加 \n：

while ((line = reader.readLine()) != null) {
    builder.append(line).append('\n');
}

这样可以尽量保留原始 XML 或 HTML 内容的文本结构。

4. Room 数据库

你学习了 Room 插入方法的返回值：

@Insert(onConflict = OnConflictStrategy.IGNORE)
long insert(FeedEntity feed);
  • 插入成功：返回新行的主键 id。
  • 因为唯一约束冲突被忽略：返回 -1。

也遇到了一个典型运行时问题：Room 默认不允许在主线程查询数据库，否则可能崩溃。

5. RecyclerView 数据刷新

收藏状态改变后，列表不会自动更新，因为 Adapter 显示的是旧的内存数据。

RecyclerView 自动改变

需要重新查询数据库，并调用 submitList() 或类似刷新方法更新 UI。

6. Activity 页面通信

详情页修改收藏状态后，使用下面的流程通知列表页刷新：

MainActivity
  startActivityForResult()

ArticleDetailActivity
  setResult(RESULT_OK, result)

MainActivity
  onActivityResult()
  refreshCurrentList()

setResult() 本身不是刷新动作，它只是准备返回结果。真正刷新发生在 onActivityResult() 中。

7. LinearLayout 布局

你学习了如何让横向按钮之间保持统一间距：

  • 少量控件：使用 layout_marginEnd。
  • 统一间隔：使用 showDividers="middle" 加透明 divider。
  • 按钮平分宽度：使用 layout_width="0dp" 和 layout_weight="1"。

二、你的薄弱之处

1. 数据对象和数据库实体的边界还不够稳

一开始容易把 ParsedFeed 和 FeedEntity 混在一起。

ParsedFeed      解析阶段的临时完整结果，包含文章列表
FeedEntity      数据库里的订阅源表对象
ArticleEntity   数据库里的文章表对象
Adapter List    UI 当前显示的数据快照

2. 异步和线程边界容易忽略

Android 中要形成固定习惯：

主线程：处理点击、更新 UI、Toast
子线程：网络请求、Room 查询和写入
Handler：把子线程结果切回主线程

3. UI 刷新责任还需要更敏感

修改数据库不等于 UI 自动更新。需要明确谁负责重新查询、谁负责提交新列表。

repository.setFavorite()
-> 数据库更新
-> setResult()
-> onActivityResult()
-> refreshCurrentList()
-> adapter.submitList()

4. 方法命名和语义一致性需要加强

例如 hasFeed() 如果实际返回的是 isEmpty()，后续逻辑会越来越绕。

命名应当让返回值含义一眼可读：

  • feedIsEmpty()：没有订阅源时返回 true。
  • hasFeeds()：有订阅源时返回 true。
  • getAllFeedArticles()：获取所有订阅源的所有文章。

三、该如何提升

1. 每写一个功能，先画数据流

例如“取消收藏后刷新列表”：

点击取消收藏
-> ArticleDetailActivity.toggleFavorite()
-> RssRepository.setFavorite()
-> ArticleDao 更新数据库
-> setResult(RESULT_OK)
-> 返回 MainActivity
-> onActivityResult()
-> refreshCurrentList()
-> 重新查询数据库
-> RecyclerView 更新

2. 重点练三个 Android 基础能力

  • Activity 之间传值和返回结果。
  • 子线程执行网络和数据库，主线程更新 UI。
  • RecyclerView Adapter 的数据刷新。

3. 读代码时按层看

UI 层：用户点了什么？
Repository 层：调用了什么业务方法？
DAO 层：查了哪张表？
Model 层：数据长什么样？
UI 层：结果如何显示？

4. 小改动后立即验证

新增功能时建议按小步推进：

加 DAO 查询
-> build
-> Repository 封装
-> build
-> UI 调用
-> build
-> 真机点击测试

每次只引入一个变量，问题会更容易定位。

5. 学会看崩溃日志

遇到崩溃时，优先在 Logcat 中找：

  • FATAL EXCEPTION
  • Caused by
  • 自己的包名：com.yosuii.rssreader

很多问题，例如主线程访问 Room，日志会直接告诉你原因。
