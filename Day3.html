今日核心收获： 你开始把 Android 功能拆成完整的数据流来看：用户手势触发事件， 后台线程执行网络和数据库任务，主线程更新 UI，再用 Git 把不同功能拆成清晰的提交。这说明你已经不只是“照着写代码”，而是在学习 App 的运行机制。

一、涉及到的知识点

1. 已读 / 未读状态的完整链路

给文章增加 isRead 字段，本质不是只加一个变量，而是要让 数据库、详情页、列表页和 Adapter 都理解这个状态。 

ArticleEntity.isRead
-> ArticleDao.setRead()
-> RssRepository.setRead()
-> ArticleDetailActivity 打开文章时标记已读
-> setResult(RESULT_OK)
-> MainActivity.refreshCurrentList()
-> ArticleAdapter 改变已读/未读样式

你也接触到了 Room 升级：新增字段后，旧数据库不能凭空知道这个列， 所以要么卸载重装，要么写 Migration(1, 2)。

2. Room Migration 的写法

Migration 的作用是告诉 Room：从旧版本数据库升级到新版本数据库时， 应该执行哪些 SQL。 

private static final Migration MIGRATION_1_2 = new Migration(1, 2) {
    @Override
    public void migrate(@NonNull SupportSQLiteDatabase database) {
        database.execSQL(
                "ALTER TABLE articles ADD COLUMN isRead INTEGER NOT NULL DEFAULT 0"
        );
    }
};

Java 的 boolean 在 SQLite 里通常以 INTEGER 保存，0 表示 false， 1 表示 true。

3. ExecutorService 为什么适合当前项目

你学习了这行代码的意义： 

private final ExecutorService executor =
        Executors.newSingleThreadExecutor();

它创建了一个只有一个后台线程的任务执行器。网络请求、Room 查询、 RSS 解析都交给它执行，避免阻塞主线程。 

主线程：点击按钮、显示 Toast、更新 RecyclerView
后台线程：网络请求、数据库读写、XML 解析
Handler：把后台结果切回主线程

4. 单线程和多线程的取舍

当前项目使用单线程不是因为多线程不好，而是因为 RSS 阅读器的任务更重视 顺序和可理解性。 

单线程：
刷新 Feed -> 查询文章 -> 更新列表

多线程：
刷新 Feed 和查询文章可能同时发生，旧数据可能覆盖新数据

对初学阶段来说，先保证数据流稳定，比追求并发速度更重要。

5. Android 中常见的后台任务方式

今天你对比了几种“不在主线程执行耗时任务”的方式：

  • Thread：最基础，适合学习原理。
  • ExecutorService：适合当前 Java/XML 项目的后台任务队列。
  • HandlerThread：适合长期存在的后台消息线程。
  • WorkManager：适合定时刷新、失败重试、后台同步。
  • Foreground Service：适合音乐播放、导航、持续下载。
  • Coroutines：适合 Kotlin 项目中的现代异步开发。

6. 下拉刷新 SwipeRefreshLayout

你给文章流增加了下拉刷新。实现方式是用 SwipeRefreshLayout 包住原来的 RecyclerView。 

SwipeRefreshLayout
  └── RecyclerView

下拉手势触发 setOnRefreshListener()， 然后根据当前文章流来源决定刷新全部订阅源，还是只刷新某一个 Feed。 

下拉刷新
-> refreshByPull()
-> repository.refreshFeed() 或 repository.refreshAllFeeds()
-> refreshCurrentList()
-> adapter.submitList()

7. 手动刷新订阅源

刷新订阅源的核心不是“重新显示列表”，而是重新走一次 RSS 数据流程： 

Feed.url
-> FeedFetcher.fetch()
-> FeedParser.parse()
-> feedDao.update()
-> articleDao.insertAll()
-> RecyclerView 重新查询并显示

文章插入使用 OnConflictStrategy.IGNORE， 可以避免重复文章覆盖已有的收藏和已读状态。

8. 聚合文章流切换来源

你把原本独立的“文章”按钮删掉，把“聚合流”升级成更统一的文章入口。 现在 selectedFeed == null 表示显示全部文章， selectedFeed != null 表示只显示某个 Feed 的文章。 

聚合文章流 · 全部
点击标题
-> AlertDialog 选择来源
-> selectedFeed = null 或某个 Feed
-> loadAllArticles()
-> 查询全部文章或指定 Feed 文章

这个设计让 UI 少了一个 tab，但文章列表的能力更集中。

9. Git 按功能拆提交

今天你还练习了把两个功能拆成两个提交，而不是把所有改动混在一起。 

feat: add pull-to-refresh for article streams
feat: switch article stream sources from title

这样以后回看历史时，可以清楚知道每个提交解决了什么问题。

二、你的薄弱之处

1. 字段升级不只是改 Entity

给 ArticleEntity 加字段以后，还要检查构造方法、DAO、 Repository、旧数据库升级和 UI 绑定。只改一个文件通常是不够的。

2. 异步任务的边界还需要继续练

你已经知道耗时任务不能放主线程，但还要继续形成固定判断： 

会不会联网？
会不会查数据库？
会不会解析大量文本？
会不会读写文件？

如果会，就优先放后台线程。

3. UI 状态和数据状态容易混在一起

例如 selectedFeed 既影响标题，也影响列表查询，还影响下拉刷新。 以后做功能时，要先想清楚“哪个变量代表当前页面状态”。

4. 功能拆提交需要有意识

一个文件里可能同时有两个功能的改动，例如 MainActivity.java 里既有下拉刷新，也有来源切换。 这时提交不能只按文件拆，而要按 diff 里的逻辑拆。

三、该如何提升

1. 每个功能先写事件流

例如“下拉刷新某个订阅源”：

用户下拉
-> SwipeRefreshLayout 触发监听
-> 判断 selectedFeed
-> 后台线程 refreshFeed()
-> 主线程关闭刷新动画
-> 重新查询文章
-> RecyclerView 更新

2. 每次新增状态都问三个问题

  • 这个状态存在哪里？内存、数据库，还是 Intent？
  • 谁会修改这个状态？
  • 状态变化后，谁负责刷新 UI？

3. 后台任务代码保持固定模板

executor.execute(() -> {
    // 后台线程：网络、数据库、解析

    mainHandler.post(() -> {
        // 主线程：更新 UI
    });
});

先把这个模板练熟，再考虑 WorkManager、协程或更复杂的异步模型。

4. 提交前先看 diff

做 Git 提交前先看： 

git status --short
git diff
git diff --cached

目标是确认这次提交只包含一个清晰功能，而不是把临时文件、 IDE 配置和其他实验改动混进去。
